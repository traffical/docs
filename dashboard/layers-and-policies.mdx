The **Layers** page is the busiest part of the dashboard. It's where every experiment, rollout, and feature-flag policy is created and managed.

  <img src="/images/placeholders/layers-list.png" alt="Layers list placeholder" />
</Frame>

## The layer list

Each row is a [layer](/concepts/layers):

- **Name** ‚Äî what you called it
- **Parameters** ‚Äî how many parameters this layer owns
- **Policies** ‚Äî how many policies are configured, with state breakdown
- **Bucket count** ‚Äî typically 10000 for the project (configurable in **Settings ‚Üí Hashing**)

The base layer is always present. Create additional layers to keep concurrent experiments cleanly separated ‚Äî group parameters that are tested together in the same layer.

## Layer detail

Clicking a layer opens its detail page with two tabs: **Parameters** (parameters in the layer) and **Policies** (policies configured against those parameters).

<Frame caption="Layer detail">
  <img src="/images/placeholders/layer-detail.png" alt="Layer detail placeholder" />
</Frame>

### Policies list

Each policy in the layer shows:

- Name and key
- Kind (static / adaptive / adaptive per-entity / adaptive contextual)
- State (draft / running / paused / completed)
- Allocations and their current bucket ranges
- Whether a rollout is attached
- Recent exposure volume

Policies are ordered by priority ‚Äî when a user is eligible for multiple, the highest-priority eligible one applies. Drag to reorder.

## Creating a policy

**New policy** opens the policy editor.

<Frame caption="Policy editor ‚Äî basics">
  <img src="/images/placeholders/policy-editor-basics.png" alt="Policy editor basics placeholder" />
</Frame>

Fill in:

1. **Name** ‚Äî human-readable
2. **Key** ‚Äî stable identifier (used in warehouse-native assignment SQL to match)
3. **Kind** ‚Äî static or adaptive
4. **State** ‚Äî start in `draft`, switch to `running` when ready

### Allocations

Add allocations ‚Äî each is a (name, bucket range, parameter overrides) tuple.

<Frame caption="Policy editor ‚Äî allocations">
  <img src="/images/placeholders/policy-editor-allocations.png" alt="Policy editor allocations placeholder" />
</Frame>

For static policies you set bucket ranges directly. For adaptive policies, start with uniform ranges ‚Äî the engine will adjust them based on observed rewards.

You can use the percentage shortcuts (50/50, 33/33/33, 90/10) or enter ranges manually. The dashboard validates that ranges don't overlap and total ‚â§ bucket count.

### Conditions (targeting)

<Frame caption="Policy editor ‚Äî conditions">
  <img src="/images/placeholders/policy-editor-conditions.png" alt="Policy editor conditions placeholder" />
</Frame>

Add condition rows to restrict eligibility. All conditions must match (AND). Operators are `eq`, `neq`, `gt`, `gte`, `lt`, `lte`, `in`, `nin`, `contains`, `regex` ‚Äî see [Policies](/concepts/policies#targeting-conditions).

Common patterns:

- `locale in [en-US, en-GB]` ‚Äî English-speaking markets
- `plan neq free` ‚Äî paying customers only
- `email contains @yourcompany.com` ‚Äî internal users only

### Eligible bucket range

For non-overlapping experiments in the same layer, set an **eligible bucket range** ‚Äî the policy only applies to users whose layer bucket falls in that range. Two policies with `[0‚Äì4999]` and `[5000‚Äì9999]` ranges never see the same user.

### Adaptive configuration

If you set Kind to `adaptive`, additional sections appear:

<Frame caption="Adaptive policy configuration">
  <img src="/images/placeholders/adaptive-config.png" alt="Adaptive configuration placeholder" />
</Frame>

- **Algorithm** ‚Äî Thompson Sampling, Epsilon-Greedy, UCB1, or Linear Contextual
- **Goal event** ‚Äî pick from existing event definitions
- **Goal type** ‚Äî `conversion_rate`, `sum`, or `count`
- **Min exposures before shift** ‚Äî prevents premature shifts on noisy early data
- **Window size** ‚Äî how often the engine re-evaluates (5m, 15m, 60m, 6h, 24h)
- **Max shift per update** ‚Äî caps how aggressively traffic can move

For per-entity bandits, expand **Entity configuration** and provide entity keys (e.g. `productId`). For contextual bandits, expand **Context logging** and list the allowed context fields.

### Rollouts

A policy can have a rollout configuration attached. Click **Add rollout** in the policy editor or on the policy detail page.

<Frame caption="Rollout configuration">
  <img src="/images/placeholders/rollout-config.png" alt="Rollout configuration placeholder" />
</Frame>

Set:

- **Target percentage** ‚Äî usually 100
- **Ramp** ‚Äî either continuous (`X% every Y minutes`) or a schedule (list of `{percentage, scheduledAt}` steps)
- **Health checks** ‚Äî one or more metric conditions evaluated before each step
- **On health violation** ‚Äî `pause`, `rollback_one_step`, `full_rollback`, or `alert_only`

See [Rollouts](/experimentation/rollouts) for the full mechanics.

## Policy detail

The policy detail page is where you read results.

<Frame caption="Policy detail ‚Äî metrics">
  <img src="/images/placeholders/policy-detail-metrics.png" alt="Policy detail metrics placeholder" />
</Frame>

Tabs:

- **Overview** ‚Äî current state, exposure counts, top-line conversion rates per allocation
- **Metrics** ‚Äî every metric attached to this policy with per-allocation results and significance
- **Allocations** ‚Äî current bucket ranges (live for adaptive policies); manual override
- **Conditions** ‚Äî targeting rules
- **Activity** ‚Äî every change to this policy (links to the [decision log](/dashboard/decisions))

### Manual actions

From the policy detail page you can:

- **Pause / resume** the policy
- **Override allocations** ‚Äî manually adjust bucket ranges (useful for cutting traffic to a misbehaving variant)
- **Promote an allocation** ‚Äî copy a winning allocation's parameter overrides into the parameter defaults
- **Duplicate** the policy ‚Äî useful for forking an experiment for a new segment
- **Complete** the policy ‚Äî final state, stops being evaluated, preserves history

All manual actions land in the [decision log](/dashboard/decisions) with the actor and timestamp.

## Tips

- **Always start in `draft`.** Build out the allocations and conditions, then flip to `running` when you're ready. Switching state is fast; building a misconfigured running policy is a panic.
- **Use the eligible bucket range** for non-overlapping experiments rather than separate layers. Layers are for parameters; ranges are for sub-segments of the same layer.
- **Name allocations descriptively.** `control` and `treatment` are conventional, but `low_price_5` is more useful than `treatment_b` six months later.
