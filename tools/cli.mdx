`@traffical/cli` is the developer CLI for config-as-code workflows. Define parameters and events in `traffical.yaml`, version-control them alongside your code, and sync with Traffical via push and pull.

## Installation

```bash
npm install -g @traffical/cli
# or run without installing
bunx @traffical/cli init
```

## Quickstart

```bash
# In your project root
traffical init

# Edit .traffical/config.yaml — add or change parameters and events
# Then push to Traffical
traffical push

# Pull dashboard changes back
traffical pull

# Check for drift
traffical status
```

## Authentication

The CLI authenticates with a server API key. Provide it in one of three ways (in order of precedence):

1. `--api-key sk_...` on the command line
2. `TRAFFICAL_API_KEY=sk_...` environment variable
3. A named profile in `~/.trafficalrc` referenced via `--profile <name>`

For CI use the environment variable. For local development, profiles let you switch between organizations and projects quickly.

## Commands

| Command | Description |
|---------|-------------|
| [`init`](#init) | Scaffold `.traffical/` and link to a project |
| [`push`](#push) | Send local config to Traffical |
| [`pull`](#pull) | Pull synced config back into the repo |
| [`sync`](#sync) | Bidirectional sync (local wins on conflict) |
| [`status`](#status) | Show drift between local and remote |
| [`import`](#import) | Add a dashboard-created parameter to local config |
| [`generate-types`](#generate-types) | Generate TypeScript types from event definitions |
| [`integrate-ai-tools`](#integrate-ai-tools) | Add Traffical context to AI tool configs (CLAUDE.md, .cursorrules, etc.) |

### `init`

```bash
traffical init [--framework react|nextjs|svelte|sveltekit|vue|nuxt|node]
               [--project <id>] [--yes] [--no-sdk-key]
```

Detects your framework from `package.json`, scaffolds `.traffical/config.yaml`, creates a public SDK key, and writes example code for the detected framework. Pass `--framework` to override detection. `--yes` runs non-interactively (useful for `bunx` flows). `--no-sdk-key` skips key creation if you already have one.

### `push`

```bash
traffical push [--dry-run] [--prune]
```

Uploads parameter and event definitions from `.traffical/config.yaml`. New parameters are created; existing synced parameters are updated.

- `--dry-run` — validate and print the diff without applying.
- `--prune` — archive synced parameters that have been removed from the file. Without this, removed parameters stay in Traffical.

Parameters managed by the CLI become **synced** in the dashboard — their definitions are read-only in the UI to prevent drift.

### `pull`

```bash
traffical pull [--include-types] [--types-output <path>]
```

Downloads synced parameters and events from Traffical into `.traffical/config.yaml`. Only synced parameters are pulled — dashboard-only parameters stay in the dashboard unless you `import` them first.

`--include-types` runs `generate-types` afterwards.

### `sync`

```bash
traffical sync [--all] [--dry-run] [--prune]
```

Bidirectional. New local parameters are pushed; new remote parameters are pulled. On conflict (both sides changed the same parameter), the local version wins. `--all` syncs every config file in the repository.

### `status`

```bash
traffical status
```

Lists three categories:

- Synced parameters that are in sync
- Synced parameters with local changes (need push)
- Dashboard-only parameters that don't exist locally (can be imported)

Exits with code `10` if drift is detected. Use this in CI:

```yaml
- name: Check Traffical drift
  run: traffical status
  env:
    TRAFFICAL_API_KEY: ${{ secrets.TRAFFICAL_API_KEY }}
```

### `import`

```bash
traffical import <key>
traffical import "ui.*"     # wildcards supported
```

Adds a dashboard-created parameter (or pattern of them) into the local config so they're managed as code from now on.

### `generate-types`

```bash
traffical generate-types [-o <path>] [-l typescript]
```

Emits TypeScript interfaces for your event definitions. Default output is `.traffical/traffical.generated.ts`. See [Type-safe events](#type-safe-events).

### `integrate-ai-tools`

```bash
traffical integrate-ai-tools
```

Adds a Traffical context block to AI tool configs in the repo: `CLAUDE.md`, `.cursor/rules`, `.github/copilot-instructions.md`, `.windsurfrules`. Useful so coding assistants know what parameters exist and how to use them.

## `traffical.yaml` schema

The config file lives at `.traffical/config.yaml`.

```yaml
version: "1.0"

project:
  id: proj_marketplace
  orgId: org_acme

parameters:
  # Main namespace — no prefix
  feature_enabled:
    type: boolean
    default: false
    description: Master toggle for the new feature

namespaces:
  checkout:
    parameters:
      button.color:                      # full key: checkout.button.color
        type: string
        default: "#1E6EFB"
        description: Primary button color on checkout
      show_trust_badges:
        type: boolean
        default: false

  pricing:
    parameters:
      discount_pct:
        type: number
        default: 0
        constraints:
          min: 0
          max: 50

events:
  purchase:
    valueType: currency
    unit: USD
    description: Completed purchase

  add_to_cart:
    valueType: count

  signup:
    valueType: boolean
```

### Parameter fields

| Field | Required | Description |
|-------|----------|-------------|
| `type` | yes | `boolean`, `string`, `number`, or `json` |
| `default` | yes | Default value (typed to match `type`) |
| `description` | no | Free-form text shown in the dashboard |
| `constraints` | no | `min`, `max`, `pattern`, `allowedValues` |

### Event fields

| Field | Required | Description |
|-------|----------|-------------|
| `valueType` | yes | `currency`, `count`, `rate`, or `boolean` |
| `unit` | no | Free-form unit (`USD`, `items`, `percent`, …) |
| `description` | no | Free-form text |

## Type-safe events

The CLI can generate TypeScript types from your event definitions. Use them to wrap your SDK's `track()` and catch invalid event names and properties at compile time.

```bash
bunx @traffical/cli generate-types
```

This writes `.traffical/traffical.generated.ts` exporting `TrafficalEventProperties` — a map from event name to the typed property shape.

Wrap your client's `track`:

```typescript
import type { TrafficalEventProperties } from "./traffical.generated";
import { createTrafficalClient } from "@traffical/node";

const traffical = await createTrafficalClient({ orgId, projectId, env, apiKey });

type TypedTrack = <E extends keyof TrafficalEventProperties>(
  event: E,
  options: { unitKey: string; properties?: TrafficalEventProperties[E]; decisionId?: string }
) => void;

export const track = traffical.track.bind(traffical) as TypedTrack;
```

```typescript
track("purchase", {
  unitKey: "user_789",
  properties: { order_total: 99.99, payment_method: "visa" },     // ✅
});

track("purchase", {
  unitKey: "user_789",
  properties: { unknown_field: true },                             // ❌ compile error
});
```

Run `generate-types` after every `pull` (or in CI as part of your build) to keep the types in sync with the dashboard.

## CI/CD integration

### Validate on pull requests

```yaml
# .github/workflows/traffical.yml
name: Traffical
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
      - run: bunx @traffical/cli status
        env:
          TRAFFICAL_API_KEY: ${{ secrets.TRAFFICAL_API_KEY }}
```

Exits with code `10` if local and remote are out of sync.

### Sync on merge

```yaml
# .github/workflows/traffical-sync.yml
name: Traffical sync
on:
  push:
    branches: [main]
    paths: [".traffical/**"]
jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
      - run: bunx @traffical/cli push --prune
        env:
          TRAFFICAL_API_KEY: ${{ secrets.TRAFFICAL_API_KEY }}
```

## Exit codes

| Code | Meaning |
|------|---------|
| `0` | Success |
| `1` | Validation error in config |
| `2` | Authentication error |
| `3` | Network or API error |
| `10` | Drift detected (`status`) |
| `11` | Experiment needs attention |

## Global options

| Option | Description |
|--------|-------------|
| `-p, --profile <name>` | Named profile from `~/.trafficalrc` |
| `-c, --config <path>` | Custom config file path |
| `-b, --api-base <url>` | Override API endpoint (for testing) |
| `-j, --format <fmt>` | `human` (default) or `json` |
| `-n, --dry-run` | Preview without applying (push/sync) |

## Environment variables

| Variable | Description |
|----------|-------------|
| `TRAFFICAL_API_KEY` | API key |
| `TRAFFICAL_API_BASE` | API endpoint override |
